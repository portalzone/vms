# Deployment

This document covers how to run VMS in production. The project is set up to deploy on [Render](https://render.com) using Docker, but the same Docker image can run anywhere that supports containers.

---

## Local development

Follow the setup steps in the main [README](./README.md). The short version:

```bash
composer install
cp .env.example .env
php artisan key:generate
php artisan migrate --seed
php artisan serve
```

For the frontend:

```bash
cd vms-frontend/vue-project
npm install
npm run dev
```

## Environment configuration

Before deploying, make sure your `.env` file (or your host's environment variable config) has the following set correctly.

### Required

```dotenv
APP_KEY=             # generated by php artisan key:generate
APP_URL=             # your production URL, e.g. https://vms.yourdomain.com
APP_ENV=production
APP_DEBUG=false

DB_CONNECTION=mysql
DB_HOST=             # your database host
DB_PORT=3306
DB_DATABASE=         # database name
DB_USERNAME=         # database user
DB_PASSWORD=         # database password
```

### Important for auth

```dotenv
# Add your frontend's origin so Sanctum allows cross-origin token requests
SANCTUM_STATEFUL_DOMAINS=your-frontend-domain.com
```

### Optional but recommended

```dotenv
LOG_CHANNEL=stack
LOG_LEVEL=warning    # avoid flooding logs with debug noise in production

MAIL_MAILER=smtp     # configure if you want email notifications
MAIL_HOST=
MAIL_PORT=587
MAIL_USERNAME=
MAIL_PASSWORD=
MAIL_FROM_ADDRESS=noreply@yourdomain.com
```

## Deploying to Render

### What's included

The project ships with everything Render needs:

- `Dockerfile` — builds a single container running PHP-FPM + nginx via supervisord
- `.render/nginx.conf` — nginx server config pointing to Laravel's `public/` directory
- `.render/supervisord.conf` — runs both php-fpm and nginx as managed processes
- `scripts/00-laravel-deploy.sh` — runs migrations and caches config on each deploy
- `start.sh` — the container's entrypoint

### First-time setup

1. Push your repository to GitHub (or GitLab).
2. In the Render dashboard, create a new **Web Service** and connect your repo.
3. Set the **Environment** to `Docker`.
4. Add your environment variables in the Render dashboard under **Environment** → **Add Environment Variable**. You need at minimum:
   - `APP_KEY`
   - `APP_URL`
   - `DB_CONNECTION`, `DB_HOST`, `DB_PORT`, `DB_DATABASE`, `DB_USERNAME`, `DB_PASSWORD`
5. If you're using a managed database, create a Render PostgreSQL or MySQL instance first and copy the connection details.
6. Hit **Deploy**.

Render will build the Docker image, run the deploy script (`00-laravel-deploy.sh`), and start the container. Subsequent pushes to your main branch trigger automatic redeploys.

### The deploy script

`scripts/00-laravel-deploy.sh` runs automatically during each deploy and handles:

```bash
php artisan migrate --force    # runs any pending migrations
php artisan config:cache       # caches config for performance
php artisan route:cache        # caches routes
php artisan view:cache         # caches compiled Blade views
php artisan storage:link       # creates the public storage symlink
```

The `--force` flag on migrate is required because Laravel won't run migrations in production without it.

## Deploying the frontend

The Vue frontend is a static site after building. Options:

### Option 1 — Render Static Site

1. Create a new **Static Site** on Render.
2. Set the **Root Directory** to `vms-frontend/vue-project`.
3. Set the **Build Command** to `npm install && npm run build`.
4. Set the **Publish Directory** to `dist`.
5. Add an environment variable `VITE_API_BASE_URL=https://your-backend-url.com/api` (if your frontend reads this from env).

### Option 2 — Any CDN / static host

```bash
cd vms-frontend/vue-project
npm run build
```

Upload the `dist/` folder to Netlify, Vercel, Cloudflare Pages, or any static file host.

### Option 3 — Serve from the same Docker container

Copy the built frontend files into Laravel's `public/` directory and let nginx serve them. This only makes sense if you want a single container deployment. It isn't the default setup but is straightforward to add to the Dockerfile.

## Database

### Local

The default `.env.example` uses SQLite. The database file is created automatically at `database/database.sqlite` the first time you run migrations.

```bash
touch database/database.sqlite
php artisan migrate --seed
```

### Production

Switch to MySQL or PostgreSQL. Create the database manually, then let the deploy script handle the schema:

```bash
# on your server / via Render shell
php artisan migrate --force
php artisan db:seed --force   # only if you want the default seeded accounts
```

> Be careful running seeders in production if the database already has real data — the `UserSeeder` uses `firstOrCreate` so it won't overwrite existing records, but it's still worth double-checking.

## Troubleshooting

**500 errors on fresh deploy**
- Check that `APP_KEY` is set.
- Make sure `storage/` and `bootstrap/cache/` are writable. In the Docker container these are writable by default.
- Check the Laravel log: `storage/logs/laravel.log`.

**CORS errors from the frontend**
- Confirm `SANCTUM_STATEFUL_DOMAINS` includes your frontend's domain (without `https://`).
- Check `config/cors.php` — the default config allows all origins in development but you may want to tighten this in production.

**Migrations fail on deploy**
- Make sure your database credentials in the environment variables are correct.
- If you're on Render, check that your database service is in the same region as the web service.
